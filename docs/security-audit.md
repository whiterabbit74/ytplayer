# Отчет по аудиту безопасности MusicPlay

Дата: 2026-03-11

## Краткое резюме
Критических уязвимостей не обнаружено. Найдено несколько важных рисков конфигурации и архитектуры: слабая защита токена (секрет по умолчанию), отсутствие ограничений CORS, отсутствие rate‑limiting, отсутствие защиты от CSRF при cookie‑аутентификации, длительный срок жизни JWT, а также потенциальные риски DoS по памяти для некоторых эндпойнтов.

## Методика
- Статический обзор серверного и клиентского кода.
- Поиск секретов и опасных паттернов.
- Проверка конфигураций безопасности (cookies, CORS, auth).
- Аудит зависимостей через `npm audit --omit=dev`.

## Находки

### Высокий риск
1. **JWT_SECRET имеет дефолтное значение**
   - Файл: `server/src/middleware/auth.ts`
   - Риск: при запуске без переменной окружения используется `musicplay-dev-secret`, что позволяет подделывать токены.

### Средний риск
1. **CORS разрешен для любого origin при использовании cookie**
   - Файл: `server/src/index.ts`
   - Риск: `cors({ origin: true, credentials: true })` позволяет браузеру отправлять cookie на API с любого origin (при наличии access‑контроля браузера и авторизации пользователя), что повышает риск злоупотреблений.

2. **Cookie‑аутентификация без защиты от CSRF**
   - Файлы: `server/src/middleware/auth.ts`, `server/src/routes/*`
   - Риск: `sameSite: "lax"` снижает риск, но не исключает CSRF для некоторых сценариев (например, POST через top‑level navigation в браузерах/клиентах). Нет CSRF‑токена или проверки Origin/Referer.

3. **Длительный TTL JWT (365 дней) и "sliding" продление**
   - Файл: `server/src/middleware/auth.ts`, `server/src/routes/auth.ts`
   - Риск: украденный токен живет слишком долго и продлевается на каждый запрос.

### Низкий риск
1. **Потенциальные DoS по памяти на stream‑буферы**
   - Файл: `server/src/routes/stream.ts`
   - Риск: MAX_BUFFER ограничен 50MB на поток, но нет ограничения общего числа одновременных потоков на пользователя/IP.

2. **Отсутствие валидации размеров входных данных для playlist/player‑state**
   - Файлы: `server/src/routes/playlists.ts`, `server/src/routes/player-state.ts`
   - Риск: можно записать большие JSON‑структуры (очередь/треки), увеличивая нагрузку на БД и память.

3. **Команда создания пользователя принимает пароль через CLI‑аргументы**
   - Файл: `server/src/create-user.ts`
   - Риск: пароль может попасть в историю shell или в список процессов.

## Рекомендации

### Обязательные (высокий приоритет)
1. **Убрать дефолтный JWT_SECRET**
   - Требовать переменную окружения и падать при отсутствии.
   - Для dev использовать отдельный `.env` (уже игнорируется в `.gitignore`).

### Рекомендуемые (средний приоритет)
1. **Ограничить CORS списком доверенных origin**
   - Явный allowlist в `cors()` (например, из `process.env.CORS_ORIGINS`).

2. **Добавить защиту от CSRF**
   - Варианты: CSRF‑токен (double‑submit), проверка Origin/Referer для state‑changing запросов, или перейти на Authorization Bearer‑токены без cookie.

3. **Сократить TTL JWT и отказаться от бесконечного продления**
   - Рекомендуется 7–30 дней с refresh‑механизмом.

4. **Добавить rate‑limiting**
   - Для /api/auth/login, /api/search, /api/stream, /api/thumb.

### Улучшения (низкий приоритет)
1. **Ограничить размер очереди/плейлистов**
   - Лимиты на количество треков и размер JSON.

2. **Ограничить число одновременных stream‑буферов**
   - Лимит по пользователю/IP, глобальный лимит.

3. **Улучшить create‑user**
   - Принимать пароль через stdin или интерактивно.

## Результаты проверки зависимостей
`npm audit --omit=dev` — уязвимостей не обнаружено.

## Следующие шаги
Если хочешь, могу:
1. Внести правки в код (CORS allowlist, JWT_SECRET обязателен, CSRF защита, rate‑limiting).
2. Подготовить конфиг `.env.example` с безопасными настройками.
3. Добавить тест‑план для безопасности.
