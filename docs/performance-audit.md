# Отчет по производительности и отзывчивости MusicPlay

Дата: 2026-03-11

## Краткое резюме
Основные узкие места связаны с:
- лишними сетевыми запросами (многоразовые загрузки плейлистов, частая синхронизация состояния),
- перерасчетом и ререндером больших списков (O(n^2) операции на списках, отсутствие виртуализации),
- слишком частыми UI‑обновлениями (тайм‑апдейты аудио триггерят ререндер всего приложения),
- отсутствием базовых ограничений на размер данных и кеширования подсказок поиска,
- неэффективной обработкой буферов стрима на сервере.

Ниже — конкретные точки в коде и рекомендации.

## Методика
- Статический анализ клиента и сервера.
- Поиск горячих путей рендера и лишних запросов.
- Оценка сетевых/БД операций и буферизации стрима.

## Находки и рекомендации

### 1) Критичные для отзывчивости UI

**1.1. Ререндер всего `App` при каждом `timeupdate` аудио**
- Файл: `client/src/hooks/useAudio.ts`, `client/src/App.tsx`
- Сейчас `useAudio` хранит `currentTime`/`duration` в состоянии, что приводит к частым ререндерам `App` (и всей ветки) 4–10 раз в секунду.
- Рекомендации:
  - Вынести UI‑состояние аудио в отдельный store (Zustand) и подписывать только нужные компоненты.
  - Либо перенести `useAudio` в компоненты `Player/MiniPlayer/FullscreenPlayer` и мемоизировать остальную часть дерева.
  - Троттлить UI‑обновления `currentTime` до 200–500 мс.

**1.2. O(n^2) поиск в `TrackList`**
- Файл: `client/src/components/TrackList.tsx`
- Для каждого трека вызывается `queue.findIndex`, что дает квадратичную сложность при больших списках.
- Рекомендации:
  - Сформировать `Map<id, index>` или `Set` один раз (через `useMemo`) и использовать O(1) lookup.

**1.3. Множественные `loadPlaylists()` в списках**
- Файлы: `client/src/components/TrackList.tsx` (в `TrackPlaylistMenu`), `client/src/components/Player.tsx`, `client/src/components/MiniPlayer.tsx`, `client/src/components/FullscreenPlayer.tsx`, `client/src/components/PlaylistList.tsx`
- `TrackPlaylistMenu` рендерится на каждый трек и вызывает `loadPlaylists()` — потенциально десятки запросов.
- Рекомендации:
  - Глобально загружать плейлисты один раз (например, в верхнем уровне после логина).
  - Добавить дедупликацию в store: `loadPlaylists` должен возвращать общий `inFlight` промис и/или проверять `loadedAt`.

**1.4. Нет виртуализации длинных списков**
- Файлы: `client/src/components/TrackList.tsx`, `client/src/components/PlaylistDetail.tsx`, `client/src/components/Queue.tsx` (если есть), `client/src/components/FavoritesList.tsx`
- Для больших списков — тяжёлый DOM и лаги.
- Рекомендации:
  - Использовать `react-window`/`react-virtual` для списков.
  - Либо применить `content-visibility: auto` и `contain-intrinsic-size` как быстрый шаг.

### 2) Лишние сетевые запросы и синхронизация

**2.1. Частая синхронизация состояния плеера**
- Файл: `client/src/hooks/usePlayerSync.ts`
- Синхронизация происходит: по таймеру 30s, на `visibilitychange`, на `beforeunload`, плюс на изменения store с интервалом 2s.
- Рекомендации:
  - Ввести батчинг и дебаунс (например, 5–10s) с «последним событием».
  - Исключить лишние триггеры при небольших изменениях (например, `position` можно синкать реже).

**2.2. Подсказки поиска без кеша и без отмены запросов**
- Файлы: `client/src/components/SearchBar.tsx`, `server/src/routes/search.ts`
- Ввод пользователя может инициировать несколько запросов, ответы приходят не по порядку.
- Рекомендации:
  - Использовать `AbortController` для отмены предыдущих запросов.
  - Ввести небольшой кеш подсказок на клиенте (Map с TTL) и/или на сервере.

### 3) Сервер: стриминг и буферы

**3.1. `Buffer.concat` на каждом cache hit**
- Файл: `server/src/routes/stream.ts`
- `slice()` вызывает `Buffer.concat(this.chunks)` каждый раз, что дорого при больших буферах.
- Рекомендации:
  - Хранить буфер как единый `Buffer` (grow‑buffer) или поддерживать структуру с индексами без конкатенации.
  - Рассмотреть ring‑buffer для диапазонов.

**3.2. Отсутствие глобальных лимитов на буферы**
- Файл: `server/src/routes/stream.ts`
- При большом числе клиентов суммарная память может расти.
- Рекомендации:
  - Лимит по пользователю/IP и глобальный лимит на количество активных буферов.
  - При превышении — LRU‑эвикшен.

### 4) База данных и запросы

**4.1. Отсутствуют индексы на частые запросы**
- Файл: `server/src/db.ts`
- Частые фильтры по `user_id`, `playlist_id`, `position`.
- Рекомендации:
  - Добавить индексы:
    - `playlists(user_id)`
    - `playlist_tracks(playlist_id, position)`
    - `favorites(user_id)`

**4.2. Нет пагинации на плейлистах/избранном**
- Файлы: `server/src/routes/playlists.ts`, `server/src/routes/favorites.ts`
- При росте данных запросы будут тяжелее и дольше.
- Рекомендации:
  - Добавить пагинацию и лимиты.

### 5) Клиент: UX‑отзывчивость

**5.1. Поиск блокирует UI при тяжелых обновлениях**
- Файл: `client/src/App.tsx`
- Большие результаты могут тормозить ввод.
- Рекомендации:
  - Использовать `startTransition` при обновлении списка результатов.
  - Показать скелетон/placeholder без блокировки ввода.

**5.2. Нет lazy‑loading для изображений**
- Файлы: `client/src/components/TrackList.tsx`, `client/src/components/PlaylistDetail.tsx`, `client/src/components/Player.tsx`
- Рекомендации:
  - Добавить `loading="lazy"` на `<img>` в списках.

### 6) Статика и сеть

**6.1. Нет gzip/brotli для API и статики**
- Файл: `server/src/index.ts`
- Рекомендация: добавить `compression` для JSON и статических файлов.

## Быстрые победы (1–2 часа)
1. Дедупликация `loadPlaylists()` в store + единичная загрузка после логина.
2. `Map`/`Set` для `queueIndex` в `TrackList`.
3. Троттлинг `currentTime` обновлений в `useAudio`.
4. Добавить `loading="lazy"` для list‑картинок.

## Средний срок (1–2 дня)
1. Виртуализация длинных списков.
2. Оптимизация буферов `stream` без `Buffer.concat`.
3. Индексы в SQLite и пагинация.

## Долгий срок (3–7 дней)
1. Перенос состояния плеера в выделенный store с минимальными подписками.
2. Пересмотр стратегии синхронизации плеера с сервером (умный батчинг).
3. Кеш подсказок/результатов поиска на клиенте и/или сервере.

## Примечания по измерениям
Для точной картины стоит добавить:
- `web-vitals` (CLS/LCP/INP) на клиенте.
- Простые тайминги для API (например, `res.locals.startTime`).
- Логи времени ответа по ключевым маршрутам `/api/search`, `/api/stream`, `/api/playlists`.
