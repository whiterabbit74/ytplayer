## P0 — немедленно (критично/высокий риск)

1. Убрать дефолтный `JWT_SECRET`
- Цель: исключить возможность подделки токенов при ошибочной конфигурации.
- Где: `server/src/middleware/auth.ts`
- Шаги:
  - Удалить значение по умолчанию `musicplay-dev-secret`.
  - При отсутствии `process.env.JWT_SECRET` выбрасывать ошибку при старте.
  - Добавить проверку на минимальную длину секрета (например, >= 32 символов).
- Готово когда:
  - Сервер не стартует без валидного `JWT_SECRET`.

2. Добавить `.env.example` и обновить README
- Цель: минимизировать ошибки конфигурации в dev/prod.
- Где: `README.md`, новый файл `.env.example`.
- Шаги:
  - Добавить `JWT_SECRET`, `YOUTUBE_API_KEY`, `CORS_ORIGINS`, `DB_PATH`.
  - Обновить раздел конфигурации в README.
- Готово когда:
  - Есть явные инструкции и шаблон.

## P1 — средний риск

3. Ограничить CORS allowlist
- Цель: запретить доступ с произвольных origin при cookie‑аутентификации.
- Где: `server/src/index.ts`
- Шаги:
  - Добавить `CORS_ORIGINS` (comma‑separated).
  - Настроить `cors({ origin: (origin, cb) => allowlist.includes(origin) })`.
  - В dev разрешить `http://localhost:*`.
- Готово когда:
  - CORS принимает только явные origin.

4. Защита от CSRF
- Цель: предотвратить подделку запросов при cookie‑аутентификации.
- Где: `server/src/index.ts`, `server/src/middleware/*`, `client/src/lib/api.ts`.
- Шаги (варианты):
  - Вариант A: double‑submit CSRF cookie + header.
  - Вариант B: проверка `Origin/Referer` для state‑changing запросов.
  - Выбрать один вариант и реализовать.
- Готово когда:
  - POST/PUT/DELETE требуют CSRF токен или валидный Origin.

5. Сократить TTL JWT и отказаться от бесконечного продления
- Цель: снизить ущерб от компрометации токена.
- Где: `server/src/middleware/auth.ts`, `server/src/routes/auth.ts`
- Шаги:
  - Уменьшить `expiresIn` (например, 7–30 дней).
  - Пересмотреть sliding‑expiration: обновлять токен не на каждый запрос, а раз в N часов.
- Готово когда:
  - Токен имеет разумный срок жизни и не обновляется на каждый запрос.

6. Rate‑limiting для чувствительных эндпойнтов
- Цель: защита от brute‑force и DoS.
- Где: `server/src/index.ts`
- Шаги:
  - Установить `express-rate-limit`.
  - Ограничить `/api/auth/login`, `/api/search`, `/api/stream`, `/api/thumb`.
  - Задать разные лимиты для auth и медиа.
- Готово когда:
  - Лимиты работают и дают 429 при превышении.

## P2 — низкий риск / улучшения

7. Ограничить размер данных для `player_state` и плейлистов
- Цель: предотвратить переполнение БД/памяти.
- Где: `server/src/routes/player-state.ts`, `server/src/routes/playlists.ts`
- Шаги:
  - Добавить server‑side валидацию размера `queue`.
  - Ввести лимит количества треков в плейлисте.
- Готово когда:
  - Сервер возвращает 400 при превышении лимита.

8. Ограничить число активных стрим‑буферов
- Цель: защититься от DoS по памяти.
- Где: `server/src/routes/stream.ts`
- Шаги:
  - Ввести глобальный лимит активных буферов.
  - LRU‑эвикшен по времени/использованию.
- Готово когда:
  - При перегрузке старые буферы удаляются.

9. Улучшить создание пользователя
- Цель: убрать утечку паролей через shell history.
- Где: `server/src/create-user.ts`
- Шаги:
  - Считать пароль из `stdin` или через интерактивный prompt.
- Готово когда:
  - Пароль не передается как CLI‑аргумент.

## Тесты и проверка

- Проверить:
  - Сервер не стартует без `JWT_SECRET`.
  - CORS блокирует неизвестные origin.
  - CSRF защита работает для POST/PUT/DELETE.
  - Rate‑limiting выдаёт 429.
